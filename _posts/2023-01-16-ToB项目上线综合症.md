许多企业专注于为某些ToB行业提供解决方案，作为乙方和不同的行业甲方签署合同项目，合同中普遍承诺：

1. 乙方把实现甲方需求的软硬件产品部署到甲方的私有云环境。
2. 作为1的一部分，乙方对接甲方不同时期的遗留系统。
3. 本次项目依赖的运行环境可能是甲方先前遗留的，也可能是伴随项目计划逐步建设。包括机房、网络、第三方设备，视项目而定，也许还有指定的服务器、PC机、移动设备。
4. 项目在某个日期开始部署上线，乙方派出实施人员在甲方现场完成软硬件的安装、部署、调测，开始试用，并支持甲方在某个日期验收项目。

有时候一个ToB项目会有多个乙方供应商、多种角色、复杂的软硬件环境，以及一些定制化功能，这些因素的综合影响下，乙方稍有不慎，在崎岖坎坷的交付之路上容易翻车。

# 1. 翻车表演

## 1.1 部署困难

乙方派驻甲方现场的实施人员发现部署时困难重重。

1. 系统不是期望的型号和版本，补丁包装不上。
2. 系统蓝屏。
3. 某设备用不了。
4. 开机启动不奏效。
5. 软件运行闪退。
6. 软件连不上设备。

## 1.2. 试用故障

完成部署后的试用阶段出现更多的问题：

1. 软件操作卡顿。
2. 软件频繁弹出对话框，耽误正常使用。
3. 设备不定期掉线。
4. 某项功能，运行结果明显不符合事实。
5. 某项功能，每到夜间3点不能用，有时自动恢复，有时只能重启机器。
6. 没有承诺的XXX功能。
7. 和第三方系统对接不成功。
8. 第三方系统故障，导致我方无法正常工作。

// 这些问题不是实施人员一人能解决，离不开兄弟团队内外和友商的支援。

用户是翻车表演的资深观众，一看势头不对，投诉飞快流转到乙方的产品研发部门。

## 1.3. 产品研发的反馈

*内容五花八门。*

### 1.3.1. 部署

1. 现场系统太老，得装系统补丁包，不能联网升级？
2. 是先在A服务器上装A1、A2，再去B服务器上装B1的吗？装B1时选择了A的地址吗？
3. 驱动装不上，我也没有好办法。
4. 没见过这个情况，你是咋用的？
5. 升级前备份过配置文件嘛？
6. 没在Fedora 37 Server上试过XXX。
7. 杀过毒嘛？你那有安装手册嘛？
8. 没用过这款设备。
9. 我这跑性能测试用例没问题，现场用的XXX设备测试过性能嘛，网络质量咋样？存储带宽和IOPS咋样？

### 1.3.2. 需求

1. 没听说要有这个功能！
2. 我们没有这个功能！
3. 这个功能都没开发完，怎么跑去部署了？
4. 你要的这个功能没法和你向用户承诺的另一个功能同时使用！
5. 做XXX功能要有YYY环境和设备，现场具备吗？
6. 什么，某项功能又要改成上周那个版本？
7. 什么，要改为支持XXX厂家设备？
8. 有50个并发客户端，怎么才1台服务器，服务器插了几根网线啊？

### 1.3.3. 排查

1. 我来远程操作下。
2. 这样a、b、c操作下，把日志发回来看看。
3. 跟踪2天再现了问题，是系统时间不对导致的。为什么现场NTP同步没生效，最近网络拓扑是不是变了？新防火墙过滤了NTP数据包？
4. 还没有突破性的发现。
5. 催能解决问题，世上就没有问题了。

# 2. 上线综合症

作为乙方，如果实施项目时经常翻车，就是感染了**上线综合症**。

## 2.1. 起因

### 2.1.1. 需求不一致

用户的诉求源自业务痛点和情感期待，如何高效、低成本地转化为一个个可量化的功能，需要高质量的沟通，优雅融合内外部资源、项目管理、产品技术解决方案及依赖的基础设施。下面的现象说明甲方和乙方对于做成什么样，缺乏明确的对齐，会让企业付出显著的额外成本。

1. 用户和业务专家商讨出想要的东西，在经过多个关联方的传达和展开后变味了，直到上线才知道。
2. 用户想要的，经过实际使用产品后，改变了想法，开始修改需求、加功能，甚至影响到技术解决方案。
3. 用户想要的，存在摸棱两可，一直未能就如何落地达成共识，大家都认为先上线再说，最后双方瞠目结舌。

![先上线](../assets/img/post/first_online.jpg)

4. 有多个用户，每个用户都有自己想要的，存在矛盾，难以全面、合理地捏合到一起。
5. 产品的关键功能和产品间的互操作，做成什么样子，多方没有明确的、统一的认识，直到上线发现相互过高期待，甚至出现重大漏洞。

发生这些现象后，企业的一致做法是迅速从产品研发团队抽调人手，和实施人员组成特种兵，高效配合铺平崎岖的交付道路。但其实不管他们效率如何，这些都是意料之外的成本，不发生问题、少发生问题、只发生低成本修复的问题，才叫高明。**事后不能缺少闭环的精益思考。**

### 2.1.2. 质量不合格

这类问题最多，有的很快能解决，有的改造起来伤筋动骨。

1. 产品UI体验不好，操作拧巴，留下粗制乱造的映像。
2. 技术解决方案无法支撑现场的环境、数据、使用方式。
3. bug多得无法使用，测试不充分。
4. 运维成本高，分析现象的时间太长、效率太低。
5. 换一台设备，产品无法使用。

### 2.1.3. 环境不一致

1. 甲方、甲方运行环境的承建方和乙方，三者没对齐最终环境的详细数据，使得产品研发部门无法按照最终环境做检验。
2. 基于1，产品研发部门不知道在最终环境下需要额外做什么，做多长时间，实施人员更不知道，全体忽略这个环节，导致计划时间不足。

### 2.1.4. 实施准备不足

实施是实际检验产品的重要阶段，在项目进程的时间线上，晚于需求分析、产品设计、研发和测试。

1. 实施人员不知道这三者的匹配度：用户想要的、产品能提供的、产品依赖的环境。
2. 实施人员临到上线才拿到最终产品，没有充裕时间熟悉产品的安装、使用、排查现象等。
3. 实施人员需要处理诸多现场的事务沟通、会议等计划外的工作。
4. 实施人员缺少分析产品和软硬件环境的运维技能。
5. 实施人员的定位模糊：
  * 我在帮助用户建设一套业务系统？
  * 我是帮助企业的产品研发部门安装系统？

### 2.1.5. 项目管理不足

翻车原因不管归类到哪个技能领域，只要频繁出现了项目不可控，就是项目管理的漏洞，而不能简单的认为这是某一个角色、某一个人的事。

## 2.2. 危害

首先打破了企业既定的计划，需要重新投入资源改进项目，看得见的成本是增加了投入、耽误其他业绩目标。

有时候看不见的成本反而更大，从实施人员开始，各个角色殚精竭虑、怄气、背锅、互怼，到失去信任、躺平比下线，没有人愿意再努力把事做好，企业损失惨重。个人也一样，周旋在多个项目之间，陷入低质而重复的沟通、排查、修复，没法更高效地提升技能、知识、视野。对比他人，不提高就是后退，大家**浪费了自己最稀缺的资源：时间。**

一旦整体得不到客观的评价和显著的改善，等于向团队暗示了一种不健康的价值观，潜移默化渗透到每个人的言行习惯，这时候才着手修复就有很高的成本，**团队和个人陷于双输**。**作为精益治理的团队领导者，想要团队和每个成员一起长期向好发展，取得双赢，就得躬身入局，不能限于每个角色哪里做的不好、做局部提升，而要思考他们做成这样的背后逻辑是什么、以及预防开始。**

## 2.3. 摸底调查

凡事预则立，不预则废。要把事情做在前头，去了解团队的状况，做到心中有底。可以尝试从下面问卷开始摸底，如果你的团队成员难以回答提问，说明大家不是一盘棋，各行其事，整体在组织、流程、执行力上有所缺乏。

### 2.3.1. 给销售的问卷

1. 你知道某个项目的产品和解决方案，前期得到了用户中的业务专家的认可吗？
2. 你知道某个项目的产品和解决方案，有哪些风险吗？
3. 你知道某个产品的最新版是什么时候发布的嘛？和上个版本相比，有哪些变化？
4. 你知道每个产品的每个大功能，分别消耗了多少人力成本投入？
5. 你知道每个产品最能打动用户的是什么嘛？
6. 你知道产品部门在未来3个月，会有哪些计划嘛？

### 2.3.2. 给实施人员的问卷

1. 你知道某个项目的需求和我方的产品功能，有哪些不匹配的吗？
2. 你直到产品依赖的环境和现场的实际环境，有哪些不匹配的吗？
3. 你知道最近1个月，每个产品的最新版本是什么时候发布的嘛？和一个版本相比，有哪些变化？
4. 你最后一次安装和频繁使用过这个产品的最新版本，是什么时候？
5. 你最喜欢产品的哪些地方？
6. 你最不喜欢产品的哪些地方？
7. 你是怎么分析电脑反应慢的原因？

### 2.3.3. 给产品团队的问卷

1. 你知道每个研发小组，最近1个月为哪个产品，做了哪些工作吗？他们实现你的设计，有哪些风险？
2. 你知道最近1个月，每个产品的最新版本是什么时候发布的嘛？和一个版本相比，有哪些变化？
3. 你上次听研发说某项功能由于XXX会延期，这会影响部署嘛？
4. 你知道销售未来3个月，会有哪些合同嘛？
5. 你知道每个合同有哪些详细的需求变化？
6. 你知道实施人员和用户遇到的困难，哪些属于易用性嘛？
7. 你知道每个产品最能打动用户的是什么嘛？
8. 你知道每个产品在什么时间、由什么角色的用户，如何使用？不同的角色有什么不同的特点？
9. 你上一次和某个项目的用户进行需求沟通是什么时候？新一轮的沟通结果对现有产品和解决方案，有哪些影响？哪些是必须要做的？

### 2.3.1. 给研发团队的问卷

1. 你知道每个产品的某些版本，要在什么时间、部署在什么项目？分别有多大的使用规模？
2. 你知道某个项目要求的产品功能清单吗？
3. 你知道项目现场部署时的各项软硬件环境参数嘛？和你的环境有什么差异？
4. 你知道最近的几个项目要求的上线时间点嘛？还有多少剩余任务？有哪些困难？
5. 平心而论，你觉得目前的计划能如期完成吗？
6. 你手头工作的质量，离你期望的状态，还差哪些东西？
7. 过去三个月你在团队中学到哪些有价值的东西？

## 2.4. 治理

由于涉及到很多的不同乙方和特殊环境，首先需要一个强有力的项目管理角色

# 3. 延申

针对某些角色和技能进行专项提高，是团队治理的常见手段，也有很多参考的文章、书籍。从长远看人和环境都会变化、流失，**授人以鱼不如授团队以渔**，搭建一个具备自我运转、自我修复的组织和流程，才是根本皆治的办法。
